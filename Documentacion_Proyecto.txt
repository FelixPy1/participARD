PARTE 2: CONECTIVIDAD DEL SISTEMA
=======================================================
• Mecanismo de conexión utilizado: ODBC (Open Database Connectivity).

• Justificación: 
La aplicación de servidor está construida utilizando el micro-framework Flask de Python. Para la conectividad con SQL Server, se emplea la librería nativa de Python llamada "pyodbc", la cual actúa como un puente sobre el driver oficial de Microsoft ("ODBC Driver 17/18 for SQL Server"). Este es el método oficial más eficiente, moderno y seguro recomendado por Microsoft y la comunidad de Python para integrar entornos backend de Python con bases de datos SQL Server, ofreciendo un excelente soporte para consultas parametrizadas (previniendo SQL Injection).

PARTE 3: ESTRUCTURA DE LA BASE DE DATOS
=======================================================
[NOTA PARA EL ALUMNO: Aquí debes insertar las capturas de pantalla desde tu SQL Server Management Studio (SSMS) de los diagramas, código de triggers y stored procedures]

• Tablas y Relaciones (Modelo Lógico):
  1. tblUsuarios: Almacena los credenciales y estado del usuario (UsuarioID, FullName, Email, PasswordHash, IntentosFallidos, BloqueadoHasta, FechaExpiracionPass). Relacionada con tblRoles.
  2. tblRoles: Define la jerarquía (RolID, NombreRol).
  3. tblActividades: Núcleo de la plataforma (ActividadID, Titulo, FechaInicio, FechaCierre, Estado, etc.). Relacionada con tblInstituciones.
  4. tblInstituciones: Catálogo de instituciones asociadas a las actividades.
  5. tblAuditoria_Actividades: Mantiene un historial de los cambios (quién creó/editó qué actividad y cuándo).

• Vistas o consultas destacadas:
  - Consulta unificada de "get_activities": Realiza un JOIN entre tblActividades y tblInstituciones, junto con un OUTER APPLY hacia tblAuditoria_Actividades para obtener dinámicamente quién fue el creador original de la actividad.

• Procedimientos almacenados (Stored Procedures):
  - "sp_BorrarActividadSegura": Procedimiento utilizado para la eliminación de actividades (referenciado en app.py al realizar peticiones DELETE).

PARTE 4: ROLES Y PERMISOS
=======================================================
La estructura jerárquica de permisos está dividida en tres roles principales:

1. Administrador (Rol_Admin):
   - Lectura: Acceso total a todas las tablas e informes.
   - Escritura: Puede crear actividades, crear nuevas instituciones, y modificar la estructura del contenido.
   - Administración: Puede crear, modificar y eliminar cuentas de usuario. Puede eliminar actividades permanentemente.

2. Editor (Rol_Editor):
   - Lectura: Puede visualizar las actividades y catálogos.
   - Escritura: Puede crear, subir y editar actividades y establecer el Estado (Activa/Deshabilitada). 
   - Restricción: No tiene acceso al panel de control de usuarios. No puede alterar los roles de otras personas.

3. Estudiante (Rol_Estudiantes):
   - Lectura: Acceso restringido únicamente a las actividades cuyo Estado es "Activa".
   - Escritura: Su única escritura permitida es iniciar sesión y redirigirse a los enlaces oficiales para inscribirse a las oportunidades.


=======================================================
PARTE 5: POLÍTICA DE SEGURIDAD EN BASE DE DATOS Y BACKEND
=======================================================
Las políticas de seguridad han sido trasladadas directamente al motor de la Base de Datos (SQL Server) y el backend de Flask para una máxima robustez:

• Políticas de contraseñas seguras:
  Las contraseñas se hashean utilizando la librería "bcrypt" con una 'salt' aleatoria en el backend. Adicionalmente, la base de datos exige que el hash no esté vacío y los inicios de sesión del servidor heredan las directivas complejas de contraseñas locales (CHECK_POLICY = ON).

• Restricción de Dominio de Correo (CHECK Constraint):
  La base de datos cuenta con una restricción CK_Usuarios_Email_Gmail que rechaza de forma directa en el motor cualquier registro de usuario cuyo Email no termine en '@gmail.com', previniendo el ingreso de datos basura o no corporativos.

• Tiempo de expiración de contraseñas (90 días):
  La base de datos realiza el control temporal a través del campo FechaUltimoCambioPassword. El sistema verifica si han transcurrido 90 días o más desde el cambio; si es así, se restringe el inicio de sesión.

• Bloqueo de Cuenta Centralizado en Procedimiento Almacenado (sp_RegistrarIntentoLogin):
  - Número máximo de intentos permitidos: 3 intentos incorrectos.
  - Penalización: Al alcanzar el límite, el Procedimiento Almacenado sp_RegistrarIntentoLogin gestiona automáticamente el bloqueo, asignando BloqueadoHasta = DATEADD(minute, 3, GETDATE()) y el contador IntentosFallidos.
  - Reinicio del estado: Si el inicio de sesión es correcto, el Procedimiento Almacenado limpia instantáneamente los contadores (IntentosFallidos = 0, BloqueadoHasta = NULL), garantizando atomicidad sin consultas ad-hoc desde la aplicación.
